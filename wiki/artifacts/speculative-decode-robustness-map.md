# Speculative Decode Robustness Map

Status: draft synthesis, not a benchmark result

## Purpose

Connect the repo's current MTP/speculative-decoding sources to robustness and quality-gate discipline before any high-TPS claim depends on a draft-token fast path.

This artifact is deliberately about serving correctness and quality risk. It does not claim that EAGLE 3.1, DFlash, CoCo, or TokenSpeed move the V4 Flash frontier in this repo. A frontier win still requires a measured non-dominated tokens/s/user versus tokens/s/GPU point under the same model, workload, hardware, correctness checks, and quality gate.

## Working Model

Speculative serving paths have two coupled systems:

- Draft path: MTP heads, EAGLE-style drafters, DFlash block-diffusion drafters, or other candidate-token generators.
- Verify path: target-model execution that accepts, rejects, rolls back, or corrects drafted tokens while preserving the target distribution or the runtime's declared approximation.

The performance win comes from increasing accepted target-equivalent tokens per expensive target step. The quality and correctness risk comes from stale cache state, incorrect rollback, parser/template sensitivity, long-context drift, draft-model mismatch, or silent output/accounting differences.

## Connections

CoCo and EAGLE 3.1 point at the same broad failure class from different sides. CoCo argues that models need training signal for recovery from corrupted contexts. EAGLE 3.1 argues that drafters can lose acceptance stability under long contexts, chat-template variation, and out-of-distribution system prompts because attention drifts toward the drafter's own generated tokens.

TokenSpeed and DFlash make the systems side concrete. TokenSpeed highlights rollback state, recurrent-state ownership, and prefix-cache semantics. DFlash introduces a different drafting shape: parallel block diffusion rather than an autoregressive draft chain. Both imply that a benchmark must record draft/verify mechanics, not just final tokens/s.

For V4 Flash specifically, model-native MTP is already in the SGLang recipe space. External EAGLE 3.1 or DFlash paths are not current V4 Flash baselines unless public draft models and reproducible launch recipes exist.

## Benchmark Rules

- Keep no-spec, model-native MTP, EAGLE-style, and DFlash-style paths as separate runtime configs.
- Record whether draft tokens are generated by model-native MTP heads, an external draft model, EAGLE 3.1, DFlash, or another method.
- Pin every external draft model by repository, revision, and file paths where applicable.
- Record draft-token count, draft tree or block shape, acceptance length, acceptance rate, acceptance by position, rejected tokens, and rollback behavior.
- Record whether accepted speculative tokens are counted identically to target-generated tokens in tokens/s/user and tokens/s/GPU.
- Run the same predeclared quality gate for no-spec and speculative configs before interpreting throughput.
- Keep corrupted-context and prompt-template robustness probes separate from the first no-cache InferenceX frontier and the first agentic prefix-cache frontier until the baseline harness is stable.
- Treat external reported speedups as `faithfully cited` or `external reported` unless reproduced locally with pinned model/runtime/hardware/workload metadata.

## Robustness Probe Ideas

These are deferred report-only probes, not part of the first V4 Flash baseline:

- Long-context prompts with fixed task semantics but varied chat templates.
- Out-of-distribution system prompts that should not change the answer task.
- Cached-prefix variants with malformed tool output or partial JSON in the prefix.
- Rare-token, weird-token, and unusual-Unicode runs inserted into otherwise normal agentic context.
- Corrupted-prefix variants that compare no-spec, model-native MTP, and external speculative paths on the same prompt set.
- Acceptance-length tracking before and after prefix corruption or prompt-template perturbation.
