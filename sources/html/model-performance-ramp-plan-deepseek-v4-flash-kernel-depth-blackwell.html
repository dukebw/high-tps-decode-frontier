DeepSeek-V4-Flash kernel anatomy

Treat each model feature as a hypothesis about the kernel/runtime path. The point is not “learn DeepSeek facts”; the point is to turn a frontier model into a concrete kernel reading and benchmarking map.

Attention

Primary kernel thesis

V4 Flash is a long-context model first. Hybrid attention reduces KV/cache pressure, but it creates new kernel complexity: compressed KV pools, sparse indexer metadata, prefix-cache coherence, and attention kernels that are no longer “just dense causal FA”.

  • Study dense FA first, then sparse/compressed variants.
  • Map prefill, decode, and append separately.
  • Track online softmax, metadata prep, and KV memory traffic as distinct costs.
MoE

Secondary kernel thesis

13B active parameters makes expert compute smaller per token than the total model suggests, but routing, ragged expert batches, FP4 scales, and all-to-all/dispatch can dominate if the engine does not align with model shape.

  • Grouped GEMM: fixed N/K, variable M per expert.
  • Dispatch/combine: local vs EP, NVLink overlap, CUDA graph constraints.
  • FP4 experts: scale layout and epilogue conversion are first-class kernel details.
Serving

Production thesis

Every kernel win must survive the serving layer: OpenAI-compatible behavior, streaming, structured output, cancellation, batching, MTP/specdecode, prefix reuse, and customer workload distributions.

Blackwell B200 kernel primitive map

This is the mental model to build while reading FA4, CUTLASS, and CuTeDSL. The hard part is no longer only coalescing and occupancy; it is overlapping the full pipeline and using Blackwell’s new data paths correctly.

What to internalize

B200 learning objectives

tcgen05 / UMMAHow Blackwell replaces Hopper WGMMA paths, why accumulators move into TMEM, and how 1SM vs 2SM MMA changes launch/sync.
TMA and multicastHow GMEM→SMEM bulk copy is described, staged, synchronized, and overlapped with MMA.
TMEM lifecycleAllocation, accumulation, copy-out to RMEM, epilogue fusion, and why register pressure changes.
Block-scaled FP4/FP8Scale tensor layout, NVFP4/MXFP4 differences, and where dequantization belongs: pre-kernel, mainloop, or epilogue.
Warp specializationDMA warp, MMA warp, epilogue warp, producer/consumer barriers, and pipeline fill/drain costs.

Week-by-week detailed plan

Each day should end with an artifact. The artifact matters because it forces the model-performance operating loop: prediction → measurement → evidence → next action.

Week 1

Top-down V4 Flash autopsy

Goal: convert model architecture into a prioritized kernel hypothesis list.

D1 · Architecture map read + diagram
  • Read DeepSeek model card, API release, vLLM recipe, SGLang day-0 post.
  • Extract: attention types, MoE active/total params, precision split, context length, MTP/specdecode knobs, runtime support.
  • Draw one transformer block with all kernels and metadata paths.
Artifact: v4-flash-kernel-map.md with “known / unknown / must measure”.
D2 · Per-token performance model math
  • Build symbolic FLOP/byte equations for dense attention, SWA, compressed attention, MoE expert GEMMs, routing, logits.
  • Use variables when exact config details are missing; the skill is bottleneck prediction, not pretending precision.
  • Classify each component as likely compute-bound, memory-bound, metadata-bound, or launch/scheduling-bound.
Artifact: notebook with roofline-style table and bottleneck priors.
D3 · Runtime support reconnaissance vLLM / SGLang / TRT-LLM
  • Check what is actually supported today: V4 Flash, MTP, FP4/FP8, compressed attention, prefix cache, EP/TP.
  • Do not spend the day making the full model run unless hardware is already available.
  • Pick a proxy workload if needed: Qwen/DeepSeek-V3.2/Mixtral-like MoE plus long-context attention shapes.
Artifact: runtime support matrix and selected capstone path.
D4 · Dense attention baseline FA mental model
  • Implement naive attention, PyTorch SDPA call, and FlashAttention call for controlled shapes.
  • Sweep head dim, sequence length, prefill vs decode, causal vs non-causal.
  • Profile memory traffic and online softmax behavior.
Artifact: attention benchmark harness with plots.
D5 · Sparse/compressed attention toy V4-like
  • Build a toy SWA + compressed-KV attention benchmark: local window plus compressed coarse tokens.
  • Implement metadata/index preparation separately from the attention compute path.
  • Measure when metadata/indexing dominates and when attention compute dominates.
Artifact: toy compressed-attention microbench + one-page lessons.
D6 · MoE routing and grouped GEMM toy dispatch
  • Implement router/top-k, token grouping, ragged expert batches, scatter/combine.
  • Benchmark uniform vs skewed routing; small vs large batch; prefill vs decode shapes.
  • Compare placeholder GEMM, PyTorch grouped execution, Triton grouped GEMM, and library paths where possible.
Artifact: MoE bottleneck table: routing, grouping, GEMM, combine.
D7 · Synthesis memo prioritize
  • Rank kernel families by expected leverage for V4 Flash serving.
  • Write “what I would inspect first in a production serving trace”.
  • Choose week-2 deep dives based on evidence, not vibes.
Artifact: week1_v4_flash_kernel_hypotheses.md.
Week 2

Kernel depth ladder

Goal: build working understanding from Triton/CUDA up to CuTeDSL, CUTLASS, FA4, and Blackwell primitives.

D8 · Triton reductions and logits SIMT
  • Implement RMSNorm, LayerNorm, residual+RMSNorm, top-k/top-p/logit mask.
  • Use these to rehearse vectorized loads, reductions, register pressure, and occupancy.
  • Run Nsight Compute on at least one kernel.
Artifact: correctness tests + benchmark table + NCU notes.
D9 · Triton attention internals online softmax
  • Implement a small FlashAttention-style online softmax kernel or heavily modify an existing tutorial.
  • Vary block sizes and num warps; track L2/HBM behavior and arithmetic intensity.
  • Write exactly why the kernel is memory-bound or compute-bound in each regime.
Artifact: “online softmax invariants” note + benchmark sweeps.
D10 · CUDA C++ kernel pass control
  • Write one CUDA C++ version of a reduction/logits kernel.
  • Practice cooperative groups/CUB where appropriate, vectorized loads, shared memory, and launch bounds.
  • Compare to Triton for both speed and debuggability.
Artifact: CUDA-vs-Triton comparison memo.
D11 · CuTe layout algebra foundation
  • Work through CuTe/CuTeDSL basics: tensors, layouts, tiling, partitioning, copy atoms.
  • Take one tiny GEMM/copy example and explain every layout transformation.
  • Do not skip layout algebra: it is the language used by FA4/CUTLASS.
Artifact: annotated CuTe layout notebook.
D12 · CUTLASS grouped GEMM MoE
  • Run CUTLASS profiler or examples for dense and grouped GEMM; focus on MoE-like variable-M shapes.
  • Study epilogue fusion, grouped scheduling, persistent kernels, block-scaled types.
  • Map DeepGEMM’s “group only M-axis” idea to CUTLASS grouped kernels.
Artifact: MoE GEMM shape table + library-choice notes.
D13 · Blackwell TMA/TMEM/tcgen05 B200
  • Walk through CUTLASS/CuTe Blackwell examples: simple tcgen05 MMA, TMA, multicast, 2SM MMA, TMA epilogue.
  • Draw the data path: GMEM→TMA→SMEM, tcgen05.mma→TMEM, tcgen05.ld→RMEM, epilogue→GMEM.
  • If you have B200 access, modify one tile shape or epilogue and profile it.
Artifact: annotated Blackwell GEMM pipeline diagram.
D14 · FA4 deep read / reproduction attention peak
  • Read FA4 for algorithm/kernel co-design: non-matmul bottlenecks, softmax changes, pipeline overlap, TMEM/2CTA behavior.
  • If on B200/H100, reproduce at least a subset of FA4 vs PyTorch/FA2/cuDNN/Triton.
  • If not, produce a “what would I profile on B200?” plan with expected NCU counters.
Artifact: FA4 anatomy memo + reproduction or dry-run protocol.
Week 3

Production-style capstone

Goal: one polished investigation that shows you can move from model/workload to measured performance action.

D15 · Capstone setup target
  • Target A: DeepSeek-V4-Flash on B200/GB200 if accessible.
  • Target B: V4 Flash architecture analysis + proxy model/shape workload if full model is impractical.
  • Define ISL/OSL/concurrency grid: e.g. 128/128, 4K/256, 32K/512, plus one long-context stress case.
Artifact: benchmark plan with exact commands and acceptance criteria.
D16 · Baseline curves no optimization yet
  • Run throughput, TTFT, TPOT, p95 latency, memory, and GPU utilization sweeps.
  • Separate prefill-heavy, decode-heavy, and mixed batches.
  • Record model/runtime config and actual git SHAs.
Artifact: baseline dashboard table + plots.
D17 · Profile and diagnose Nsight
  • Use Nsight Systems for timeline gaps and CPU/GPU overlap.
  • Use Nsight Compute for one attention or GEMM kernel.
  • Pick one bottleneck: attention/KV, MoE GEMM, routing/dispatch, logits/sampling, CPU scheduling, or runtime config.
Artifact: trace screenshots + bottleneck hypothesis.
D18 · Patch path A attention/runtime
  • Try one credible intervention: attention backend, chunked prefill, prefix-cache knob, batch/concurrency shaping, FA backend, or CUDA graph setting.
  • Keep the patch small enough to attribute causality.
Artifact: patch A before/after table.
D19 · Patch path B MoE/logits
  • Try one MoE/GEMM/logits intervention: grouped-GEMM path, routing layout, logit-mask kernel, top-k sampling, or FP4 scale handling.
  • If no code patch is feasible, do a config/library swap and explain why it is the right production move.
Artifact: patch B before/after table.
D20 · Write the report evidence
  • Use the production performance-report shape: workload, hardware/runtime/config, baseline, hypothesis, profiler evidence, intervention, result, caveats, next 3 experiments.
  • Include failed hypotheses if they were evidence-producing.
Artifact: 3–5 page performance bring-up report.
D21 · Day-one performance pack handoff
  • Create a repo README, benchmark scripts, kernel ladder index, report PDF/HTML, and “questions for week 1 on a model-performance team”.
  • Write what you know well vs what context you need from the production serving stack.
Artifact: day-one pack you can reference immediately.

Quality bar

No vibes. Evidence only.

  • Every claim has one of: formula, benchmark number, profiler screenshot, config diff, or source citation.
  • Separate “real DeepSeek run” from “proxy-shape inference”.
  • Separate kernel speedup from end-to-end serving improvement.
  • Explicitly mention feature caveats: streaming, structured output, cancellation, OpenAI compatibility, prefix cache, MTP/specdecode.

Day-one model-performance questions

These are tuned for this ramp plan and your background: show you can own a performance investigation without assuming the answer is always “write a custom kernel.”

Workload

What should I reproduce first?

Which current customer/model/workload has the highest model-performance leverage: TTFT, TPOT, throughput, cost, long context, MoE, structured output, or reliability?

Runtime

Which path is strategic?

For current frontier open models, where should a serving team prefer TensorRT-LLM, SGLang, vLLM, FlashInfer, custom CUDA/Triton, or upstream patches?

Kernels

Where do custom kernels live?

What is the review bar for correctness, numerics, shapes, profiler evidence, regression tests, and production feature compatibility?

Measurement

What is the benchmark standard?

What datasets, ISL/OSL distributions, concurrency sweeps, p50/p95 metrics, hardware clock settings, and traces count as reproducible?

Infra

What should I not reinvent?

Which internal tools already exist for traces, microbenchmarks, runtime configs, deployment packaging, and regression dashboards?
